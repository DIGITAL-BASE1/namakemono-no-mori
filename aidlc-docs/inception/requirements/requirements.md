## 機能要件

### FR-001: スワイプ報告
| 項目 | 内容 |
|---|---|
| **概要** | ユーザーが日常の「ダメ」をスワイプ形式で報告する |
| **詳細** | 全30種・8カテゴリのダメカードからランダム表示。左右スワイプで「やった/やってない」を選択 |
| **入力** | スワイプ操作（左/右） |
| **出力** | ナマケコイン獲得、出現条件の進行 |
| **ビジネスルール** | 1日1回。途中離脱でもコイン獲得。1枚もスワイプせず閉じると「究極のナマケコイン」獲得 |

### FR-002: ナマケコイン経済システム
| 項目 | 内容 |
|---|---|
| **概要** | 怠惰を通貨（NMK）に変換する経済圏 |
| **獲得方法** | スワイプ「やった」1枚=5NMK、途中離脱=残り枚数×3NMK、未スワイプ閉じ=20NMK、放置ボーナス=10NMK/日（蓄積型：次回ログイン時にまとめて受け取り）、日次収穫=5〜15NMK/匹/日 |
| **使い道** | ショップでアイテム購入（おやつ10〜30NMK、おもちゃ50〜100NMK、特別な贈り物200〜500NMK） |

### FR-003: ナマケモノ出現・コレクション
| 項目 | 内容 |
|---|---|
| **概要** | ダメ報告の蓄積により条件を満たすとナマケモノが森に住み着く |
| **MVP体数** | 5体（ネムリン、モグモグ、アトデーノ、スクロン、メンドクサウルス） |
| **図鑑** | 16体分のシルエット枠（将来拡張用） |
| **出現条件** | カテゴリ別のダメ報告回数に基づく。ヒントのみ公開、条件非公開 |
| **特殊条件** | メンドクサウルスはスワイプ5回以上途中離脱で出現 |

### FR-004: 森の表示・成長
| 項目 | 内容 |
|---|---|
| **概要** | ナマケモノが増えるほど森が豊かになる |
| **段階** | 小さな森(1〜3匹)→にぎやかな森(4〜8匹)→豊かな森(9〜15匹)→伝説のナマケモノの森(16匹+) |
| **表示** | ナマケモノが森の中で思い思いに過ごしている様子を表示 |
| **つぶやき** | ナマケモノをタップするたびに一言つぶやく（各キャラ5種類）。時々前向きな言葉が混ざる |

### FR-005: ショップ・アイテム贈呈
| 項目 | 内容 |
|---|---|
| **概要** | ナマケコインでアイテムを購入し、ナマケモノに贈る |
| **効果** | 親密度UP（おやつ=小、おもちゃ=中、特別な贈り物=大） |
| **カテゴリ** | おやつ、おもちゃ、特別な贈り物 |

### FR-006: 親密度システム
| 項目 | 内容 |
|---|---|
| **概要** | ナマケモノごとに親密度を管理（★1〜★5） |
| **上昇方法** | アイテム贈呈 |
| **好き/嫌い（MVP v2）** | 各ナマケモノに好きなもの・きらいなものが裏で設定。好きなものを贈ると親密度が大きく上がり、きらいなものを贈ると微減。好き嫌いは明示されず、キャラの性格から推測する |
| **解放要素** | ★3で手紙解放、★5（MAX）でおつかい解放 |

### FR-007: 手紙コレクション
| 項目 | 内容 |
|---|---|
| **概要** | 親密度が上がるとナマケモノから手紙が届く |
| **内容** | 説教ではなく、そばにいてくれる感じの短くてあたたかい言葉 |
| **保管** | 手紙コレクションに保管、読み返し可能 |

### FR-008: 日次収穫
| 項目 | 内容 |
|---|---|
| **概要** | 森に住み着いたナマケモノが毎日ナマケコインを生む |
| **収穫量** | 5〜15 NMK/匹/日 |
| **親密度ボーナス** | 親密度が高いナマケモノほど日次収穫で生むコインが多くなる |
| **トリガー** | アプリ起動時にオンデマンド計算（EventBridge不使用） |
| **理由** | 〜5ユーザーのデモ規模のためバッチ処理の複雑さを回避。ユーザーがアプリを開いた時に「前回収穫からの経過日数」を計算して付与する方式 |

### FR-009: おつかい機能
| 項目 | 内容 |
|---|---|
| **概要** | 親密度MAXのナマケモノがLINE上で現実世界の小さな行動を頼む |
| **解禁条件** | ナマケモノの親密度MAX（★5） |
| **配信頻度** | 2〜3日に1回（ランダム間隔） |
| **配信時刻** | 10:00 / 13:00 / 16:00 のいずれか（日替わりでランダム決定） |
| **トリガー** | EventBridge × 3本（10:00、13:00、16:00）。各実行時に対象ユーザーを判定 |
| **配信場所** | LINEトーク（Messaging API） |
| **AI活用** | おつかい生成（パーソナライズ）+ 応答生成 + 達成判定（Amazon Bedrock） |
| **ガードレール** | 天気・気温（ユーザーの地域設定に基づく）、祝日・曜日、おつかい履歴を参照 |
| **地域情報** | ユーザーテーブルの地域フィールドを参照（MVP v1はデフォルト「東京」固定） |
| **段階** | Level 1(外に出る)→Level 2(少し歩く)→Level 3(新しいこと)→Level 4(誰かと関わる) |
| **未達成時** | ペナルティなし |

### FR-010: おつかい応答・会話
| 項目 | 内容 |
|---|---|
| **概要** | ユーザーがおつかい報告するとナマケモノがAIで応答する |
| **会話量** | 2〜3往復の短い会話 |
| **報告方法** | LINEトークに返信（写真・テキスト） |
| **達成判定** | AIが判定。適当な画像や無関係な返答は未達成 |
| **セキュリティ** | プロンプトインジェクション対策必須（下記NFR-007参照） |

### FR-011: 森の日記
| 項目 | 内容 |
|---|---|
| **概要** | おつかいの内容と報告がナマケモノの言葉で記録される |
| **表示場所** | アプリ内（森の日記画面） |
| **目的** | 振り返ると「自分が動いた軌跡」が残る |
| **生成方法** | Amazon Bedrockで会話ログを要約し、ナマケモノのキャラクタートーンで一文にまとめる |
| **記録タイミング** | おつかいが **達成** と判定された時のみ記録。未達成時は日記に残さない |

### FR-012: 通知
| 項目 | 内容 |
|---|---|
| **概要** | LINE Push通知でユーザーにアプリ復帰を促す |
| **種類** | 新しいナマケモノ到着通知、夜のスワイプ促し通知（ナマケモノ固有セリフ） |
| **配信** | LINE Messaging API（Push） |
| **時刻設定** | MVP v1ではデフォルト21:00固定。MVP v2で設定画面から変更可能にする |
| **開発用手動トリガーAPI** | 動画撮影時に通知タイミングを自由に制御するための開発者向けAPI |

### FR-013: 絵本オンボーディング
| 項目 | 内容 |
|---|---|
| **概要** | 初回起動時に3ページの絵本で世界観に入る |
| **ページ数** | 3ページ |
| **目的** | 世界観の導入、操作説明なし |

### FR-014: 初回体験設計
| 項目 | 内容 |
|---|---|
| **概要** | Day 1でネムリン確定出現、Day 2でモグモグ確定出現 |
| **設計意図** | 1.5日分のループ体験を確実に提供し離脱を防ぐ |
| **スワイプ設計** | 初回5枚のダメカードに「ねむねむ」「もぐもぐ」条件を必ず含める |

## 非機能要件

### NFR-001: パフォーマンス
| 項目 | 内容 |
|---|---|
| **同時接続** | 〜5ユーザー（プレゼンデモ用途、録画中心） |
| **レスポンス** | API応答 < 3秒（AI応答を除く） |
| **AI応答** | おつかい応答 < 10秒 |

### NFR-002: 可用性
| 項目 | 内容 |
|---|---|
| **レベル** | デモ用途のため高可用性は不要 |
| **インフラ** | サーバーレス（Lambda）で基本的な可用性を確保 |

### NFR-003: デプロイ
| 項目 | 内容 |
|---|---|
| **環境** | 開発環境のみ（ハッカソン用途） |
| **方法** | AWS CDKで手動デプロイ |
| **ホスティング** | S3 + CloudFront（フロントエンド）、Lambda + API Gateway（バックエンド） |

### NFR-004: テスト
| 項目 | 内容 |
|---|---|
| **レベル** | ユニットテストのみ |
| **対象** | ビジネスロジック（コイン計算、出現条件判定、親密度計算等） |

### NFR-005: 画像素材
| 項目 | 内容 |
|---|---|
| **方針** | チームの画像作成担当が作成 |
| **開発時** | プレースホルダー画像で開発し、素材完成後に差し替え |
| **格納場所** | assets/ ディレクトリ |

### NFR-006: ゲームバランスパラメータの可変性
| 項目 | 内容 |
|---|---|
| **概要** | NMK獲得量・コスト等のゲームバランスに関わる数値を、コード変更なしで容易に調整可能にする |
| **対象パラメータ** | スワイプ報告のNMK獲得量、途中離脱ボーナス係数、放置ボーナス、日次収穫量、ショップアイテム価格、親密度上昇量、出現条件の閾値 |
| **実現方法** | 設定ファイル（JSON/環境変数等）で外部化し、コード内にハードコードしない |
| **理由** | ゲームバランスは実際にプレイしながら調整が必要。開発初期から設定の外部化を前提とする |

### NFR-007: AIエージェントのセキュリティ（プロンプトインジェクション対策）
| 項目 | 内容 |
|---|---|
| **概要** | おつかい会話でユーザー入力がAIに渡される際のセキュリティ対策 |
| **対策1: 入力サニタイズ** | ユーザー入力をシステムプロンプトと明確に分離。ユーザーメッセージは「ユーザー発言」として囲い込み、システム指示と混同させない |
| **対策2: ロール固定** | AIの応答をナマケモノのキャラクター・トーンに強く制約するシステムプロンプト設計。キャラクターから逸脱する応答を抑制 |
| **対策3: 出力フィルタリング** | AI応答に不適切な内容（個人情報漏洩、有害コンテンツ、システム情報の露出）が含まれないかチェック |
| **対策4: 会話長制限** | 1回のおつかいにつき最大10往復に制限し、長時間の誘導攻撃を防止（通常の会話は2〜3往復で自然に終了する想定） |
| **対策5: Bedrock Guardrails** | Amazon Bedrock Guardrailsを活用し、有害コンテンツのフィルタリングとトピック制限を設定 |

## 技術スタック（確定）

| レイヤー | 技術 |
|---|---|
| フロントエンド | React + Vite + LIFF SDK |
| バックエンド | AWS Lambda + API Gateway |

## 配信形態

- **LINEアプリ内完結型（LIFF）**
- ユーザーはLINEの友だち追加が入口
- 森、ショップ、スワイプ報告、図鑑等の全UIはLINEアプリ内で表示
- おつかいトーク・通知はLINEトークで配信
- 認証はLIFF（LINEログイン）
