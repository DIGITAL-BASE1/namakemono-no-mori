# Unit of Work 定義 — ナマケモノの森 MVP v1

---

## チーム体制

### ユニット分割の背景
メンバーの1人が実装期間中に長期出張で不在となるため、**常駐2名でコード実装を並列に進め、リモートメンバーにはコード不要のアセット制作を担当してもらう**構成とした。これにより、物理的に離れていても非同期で貢献でき、チーム全員がMVP完成に寄与する体制を実現している。

| メンバー | 状況 | 主担当 |
|---|---|---|
| メンバーA | 常駐 | Unit 1（バックエンド+インフラ）。プレゼン資料作成も主導 |
| メンバーB | 常駐 | Unit 2（フロントエンド）。動画撮影も主導 |
| メンバーC | 長期出張（リモート） | Unit 3（アセット制作）。必要な時にタイムリーに納品。都合よくデザイン系スキルが高い |

### 開発方針
- **Unit 1とUnit 2は完全並列で開発**。待ち時間を作らない
- Unit 2はモックAPIで開発を開始し、Unit 1のAPI完成後に実APIに切り替え
- Unit 3はオンデマンドで制作。フロント開発の進捗に合わせてタイムリーに納品
- AI-DLCのCONSTRUCTION phaseでは、Functional Designを両ユニット同時に実施し、Code Generationも並列で進める

## Unit 1: バックエンド + インフラ

### 責務
- AWSインフラ構築（CDK）
- Lambda関数の実装（全API機能）
- DynamoDBテーブル設計・構築
- LINE連携（LIFF認証検証、Messaging API、Webhook）
- EventBridgeスケジューラー設定
- 開発用手動トリガーAPI

### 含まれる機能ブロック
- ユーザー管理（登録、状態取得）
- ゲームコア（スワイプ報告処理、コイン計算、出現条件判定、コイン収穫）
- ショップ・親密度（アイテム購入、贈り物、親密度更新）
- 図鑑（図鑑データ取得）
- 通知（夜のスワイプ通知、朝の到着通知、手動トリガー）
- LINE連携（Webhook受信、トークン検証）

### インフラ構成
- S3 + CloudFront（フロントエンドホスティング）
- API Gateway + Lambda
- DynamoDB
- EventBridge
- CDKスタック（dev / prod環境分離）

### 実装順序
1. **縦串スパイク**（最優先）: LIFF起動→userId取得→Lambda検証→DynamoDB保存→LINE Push送信
2. API機能の実装（ユーザー登録→スワイプ報告→コイン収穫→ショップ→贈り物→図鑑）
3. 通知スケジューラー（EventBridge→Lambda→LINE Push）
4. マスターデータJSON整備

### 成果物
- CDKスタック（全インフラ定義）
- Lambda関数群
- マスターデータJSON（ナマケモノ定義、カード定義、アイテム定義、手紙定義）
- API仕様（Functional Designで詳細化）

## Unit 2: フロントエンド

### 責務
- React + Vite + LIFF SDKによるSPAの実装
- 全画面の実装（森、スワイプ、ショップ、贈り物、図鑑、手紙）
- オンボーディングDialog
- 演出・アニメーション（召喚演出、コイン獲得演出、図鑑登録演出等）
- Unit 3のアセットの組み込み

### 含まれる画面
- ForestPage（森メイン画面）
- SwipePage（スワイプ報告）
- ShopPage（ショップ）
- GiftPage（贈り物）
- EncyclopediaPage（図鑑）
- LetterPage（手紙）
- OnboardingDialog（初回チュートリアル）

### 実装順序
1. LIFF初期化 + ルーティング基盤
2. スワイプ報告画面（コアUX）
3. 森画面（ナマケモノ表示、タップ、収穫）
4. オンボーディングDialog
5. ショップ + 贈り物画面
6. 図鑑 + 手紙画面
7. 演出・アニメーション磨き込み

### 成果物
- React SPAアプリケーション
- S3にデプロイ可能なビルド成果物

## Unit 3: アセット制作

### 責務
- ナマケモノのイラスト制作（5体）
- 森の背景イラスト（成長段階1〜2）
- ショップアイテムのアイコン
- ダメカードのカテゴリアイコン
- オンボーディング絵本のイラスト（3ページ）
- UIアイコン・装飾素材
- トンマナ（トーン＆マナー）の統一

### 制作物一覧

| カテゴリ | 内容 | 枚数目安 |
|---|---|---|
| ナマケモノ | 5体のメインイラスト + リアクション差分 | 10〜15枚 |
| 森の背景 | 成長段階1（小さな森）、段階2（にぎやかな森） | 2枚 |
| ショップアイテム | おやつ、おもちゃ、特別な贈り物 | 8〜12枚 |
| カテゴリアイコン | 8カテゴリ分 | 8枚 |
| 絵本 | オンボーディング3ページ分 | 3枚 |
| UI素材 | コインアイコン、足跡、シルエット枠、吹き出し等 | 10〜15枚 |
| **合計** | | **40〜55枚** |

### 実装順序
1. トンマナ定義（色味、タッチ、世界観の統一基準）
2. ナマケモノ5体のメインイラスト（最優先。フロントの開発に必要）
3. 森の背景（段階1, 2）
4. オンボーディング絵本
5. ショップアイテム + カテゴリアイコン
6. UI素材 + リアクション差分

### 納品形式
- PNG（透過背景）
- サイズ規格はフロントエンド担当と事前に合意
- Git経由で共有（モノレポの`assets/`ディレクトリ）

## ユニット間の連携ポイント

| 連携 | 内容 | タイミング |
|---|---|---|
| Unit 1 → Unit 2 | API仕様の共有（エンドポイント、レスポンス形式） | Functional Design完了後 |
| Unit 3 → Unit 2 | イラスト素材の納品 | 随時（優先度順に納品） |
| Unit 1 → Unit 2 | CloudFront URL、LIFF ID等の環境情報 | 縦串スパイク完了後 |
| Unit 2 → Unit 3 | 画像サイズ・形式の仕様 | 開発初日に合意 |
