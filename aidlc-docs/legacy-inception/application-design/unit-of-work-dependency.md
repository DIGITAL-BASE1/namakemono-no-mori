# Unit of Work 依存関係 — ナマケモノの森 MVP v1

---

## 依存関係マトリクス

| | Unit 1 (Backend+Infra) | Unit 2 (Frontend) | Unit 3 (Assets) |
|---|---|---|---|
| **Unit 1** | — | API仕様を提供 | なし |
| **Unit 2** | APIを呼び出す | — | イラスト素材を使用 |
| **Unit 3** | なし | 画像サイズ仕様を受ける | — |

---

## 依存の方向

```
Unit 3 (Assets)
    │
    │ イラスト素材を納品
    ▼
Unit 2 (Frontend) ←──── API仕様 ──── Unit 1 (Backend+Infra)
    │                                      │
    │ API呼び出し                           │
    └──────────────────────────────────────→┘
```

---

## 依存によるリスクと対策

| 依存関係 | 対策（待ち時間を作らない） |
|---|---|
| Unit 2がUnit 1のAPIを呼ぶ | Unit 2はモックAPI（MSW）で開発。Day 8に実APIに切り替え |
| Unit 2がUnit 3のイラストを使う | プレースホルダー画像で開発。イラストは完成次第差し替え |
| Unit 1の縦串スパイクが失敗した場合 | 最初の1〜2日で判明。失敗時はWeb Push or 疑似通知に方針転換 |

---

## 実装タイムライン（2週間 = 14日）

### 全体スケジュール

| 期間 | フェーズ | 内容 |
|---|---|---|
| Day 1〜2 | 立ち上げ | 縦串スパイク + 基盤構築 + API仕様合意 |
| Day 3〜7 | 開発（前半） | API実装 + 画面実装（並列） |
| Day 8〜9 | 開発（後半） | 結合 + 演出磨き + バグ修正 |
| Day 10〜12 | プレゼン準備 | 動画撮影 + スライド作成 |
| Day 13〜14 | 仕上げ | リハーサル + 微調整 + 提出 |

| 4 | 出現条件判定エンジン + コイン収穫API |
| 5 | ショップ購入API + 贈り物API + 親密度計算 |
| 6 | 図鑑API + 手紙解放判定 + マスターデータJSON整備 |
| 7 | 通知Lambda（夜のスワイプ通知 + 朝の到着通知）+ EventBridge設定 + 手動トリガーAPI |
| 8 | Unit 2との結合テスト。モックAPI→実API切り替え支援。バグ修正 |
| 9 | 結合テスト続き。動画撮影用のデータセットアップ（Day1〜3の状態を再現するシード） |
| 10 | 動画撮影サポート（通知の手動トリガー、データリセット等） |
| 11 | プレゼン資料作成（スライド構成、設計思想パート） |
| 12 | プレゼン資料作成（動画埋め込み、将来ビジョンパート） |
| 13 | リハーサル。フィードバック反映 |
| 14 | 最終調整 + 提出 |

### 並列開発を成立させるための合意事項（Day 1に決める）

| 項目 | 内容 |
|---|---|
| API仕様 | リクエスト/レスポンスのJSON形式、エンドポイントURL規則、エラー形式、認証ヘッダー |
| 画像仕様 | ナマケモノ: PNG透過 512x512、背景: 画面幅対応、アイテム: 128x128、アイコン: 64x64 |
| モックAPI | MSW（Mock Service Worker）でフロント側にモックを用意。API仕様に準拠したレスポンスを返す |
| Git運用 | mainブランチ保護。feature/unit1-*, feature/unit2-* でブランチ分け。Day 8に結合 |
