## 分解方針の確認質問

### Q1: ユニット分割の粒度

Application Designでは「フロントエンド」「バックエンド」「インフラ」の3レイヤーがあります。ユニットの分割方針について：

- A) レイヤー分割（フロントエンド / バックエンド / インフラ）— 3ユニット。チームメンバーの得意分野で分担しやすい
- B) 機能ドメイン分割（ゲームコア / 通知・LINE連携 / フロントエンド）— ドメイン知識で分割
- C) 縦串分割（LIFF縦串スパイク / コアゲーム機能 / 演出・UI）— 最初に技術リスクを潰す構成
- D) その他（自由記述）

[Answer]:D 実はメンバーの一人が実装期間中の大半で長期出張となるため不在です。でも基本的にはA

### Q2: 開発の優先順位

execution-plan.mdのリスク対策で「LIFF縦串スパイクを最初に」と記載しています。ユニットの実装順序について：

- A) 縦串スパイク → バックエンド → フロントエンド → 結合（ウォーターフォール的）
- B) 縦串スパイク → 全ユニット並行開発（チームメンバーで分担）
- C) 縦串スパイクをUnit 1として独立させ、完了後に残りを並行
- D) その他（自由記述）

[Answer]: D B, Cって何が違うの？

### Q3: チーム分担

2〜3人チームでの分担を想定しています。ユニット分割はチーム分担と一致させますか？

- A) はい。1人1ユニットを担当する形にしたい
- B) いいえ。ユニットは論理的な分割で、担当は柔軟に決める
- C) 部分的に一致。コアは1人が見て、残りを分担
- D) その他（自由記述）

[Answer]:C

### Q4: インフラユニットの扱い

CDKによるインフラ構築（S3, CloudFront, API Gateway, Lambda, DynamoDB, EventBridge）は：

- A) 独立したユニットとして最初に構築。他のユニットはインフラが整ってから開発
- B) バックエンドユニットに含める。バックエンド担当がインフラも管理
- C) 全ユニットに分散。各ユニットが自分のインフラを定義（CDKスタック分割）
- D) その他（自由記述）

[Answer]:B

### Q5: コード構成（モノレポ vs マルチレポ）

- A) モノレポ（1つのリポジトリに全コード）— 小規模チームに最適。依存管理が楽
- B) マルチレポ（フロント/バックエンド/インフラで別リポジトリ）— 独立性高い
- C) モノレポ + パッケージ分割（packages/frontend, packages/backend, packages/infra）
- D) その他（自由記述）

[Answer]:A
